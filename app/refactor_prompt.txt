Action Type Mapping (What triggers what):
When replacing code in specific functions, use these exact action_type values to route to the next person in line:
1. EXPENSE MODULE (module=models.NotificationModule.EXP)
Inside create_expense (if not draft) OR submit_expense: The next action is -> models.ProjectActionType.EXP_APPROVE_L1

2. BC MODULE (module=models.NotificationModule.BC)
Inside submit_bc: The next action is -> models.ProjectActionType.BC_APPROVE_L1
Inside approve_bc_l1: The next action is -> models.ProjectActionType.BC_APPROVE_L2
Inside approve_bc_l2: The next action is -> models.ProjectActionType.BC_CONFIRM (To notify the SBC)
3. FUNDS/CAISSE MODULE (module=models.NotificationModule.CAISSE)
Inside create_fund_request: The next action is -> models.ProjectActionType.FUND_APPROVE

4. ACCEPTANCE MODULE (module=models.NotificationModule.ACCEPTANCE)

Inside approve_act_rqc: The next action is -> models.ProjectActionType.ACT_APPROVE_PM
Inside approve_act_pm: The next action is -> models.ProjectActionType.ACT_APPROVE_PD
EXCEPTION: "Downward" or "Final" Notifications
If an action is a REJECTION (e.g., reject_bc, reject_expense) or an ACKNOWLEDGEMENT (e.g., final receipt of funds), DO NOT use trigger_workflow_notification.
For rejections and final closures, the notification goes directly to the original Creator (creator_id or requester_id). Leave the direct create_notification and direct email calls intact for these downward flows, as they do not use the forward-moving Matrix.
